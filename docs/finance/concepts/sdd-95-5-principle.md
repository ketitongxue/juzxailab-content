# SDD 的 95/5 原则

## 核心观点

Spec-Driven Development（SDD，规格驱动开发）的价值分布极不均衡：**约 95% 的价值来自动手前先把需求想清楚、写成结构化文档交给 AI；只有约 5% 的价值取决于具体用哪个工具**（OpenSpec / Spec-Kit / Superpowers / BMAD 等，任选其一皆可）。

常见误区是把时间分配反了：花大量时间研究、对比、安装工具，却在实际编写规格时觉得麻烦而放弃，回到无规格的 "Vibe Coding"。这是"逛工具市场"而非真正实践 SDD。

## 朴素的核心习惯

95% 的价值可由一个朴素习惯获得：动手前先写约四句话级别的结构化规格。建议模板（`specs/<功能名>.md`）包含四个二级标题，每个标题下填 3–5 条：

1. **要做什么** —— 明确范围与目标
2. **不做什么** —— 划定边界、排除项
3. **边界和验收** —— 成功标准与约束
4. **怎么验证** —— 验证方式与测试思路

写代码前先把这份 md 贴给 AI。无需安装任何工具即可获得绝大部分价值。

## 工具的引入时机

那 5% 的工具价值，应在养成"动手前先写规格"的习惯之后再考虑。判断信号：当重复编写规格感到疲劳、希望有工具自动推进重复步骤时再引入。届时对工具的提问方式也会从"哪个工具好"转变为"这个工具能替我省掉哪一步重复动作"。

## 方法论启示

- 越朴素越管用：四个二级标题本身即是核心价值；过度复杂化反而被工具稀释。
- 核心箴言：**不要在 5% 的事情上花 95% 的时间。**
- "先定义验收标准再实现"的思路，与回测验证、模型评估等量化研究中的质量控制同源——先约定"怎样算正确"，再去判断结果。
